* 起動シーケンス

** 3フェーズの全体像

Emacs の起動は内部的に 3 段階に分かれています。

- *Phase 1 (early-init.el)* — フレームが描画される前の最初期処理。
  =package.el= を無効化して、ディレクトリを準備して、GC を一時的に緩めます。
- *Phase 2 (init.el)* — =straight.el= と =leaf= をブートストラップして、
  モジュールシステムを起動する手前の準備をします。
- *Phase 3 (modules.el)* — =my:modules= と =my:modules-extra= を連結した順序でロードします。
  1 本失敗しても残りは続きます（delete-safety）。

[[file:svg/01_boot_flow.svg]]

** early-init.el の主なセクション

| セクション | 内容 | コード例 |
|------------+------+---------|
| §1 | =package.el= 無効化 | =(setq package-enable-at-startup nil)= |
| §2–3 | ベースディレクトリ + straight ベース | =my:d= / =.cache/build-32.0= |
| §4 | 組み込み疑似パッケージ | =straight-built-in-pseudo-packages= に =org= を追加 |
| §5 | macOS Homebrew ツールチェーン | =HOMEBREW_PREFIX= 探索 |
| §6 | native-comp | =eln-cache= パス、=native-comp-jit-compilation= |
| §7 | =no-littering= 互換変数の事前設定 | =no-littering-var-directory= |
| §8 | GC ガード | =file-name-handler-alist= を nil に退避 |
| §9 | バックアップポリシー | =version-control t=、=kept-new-versions 6= |
| §10 | 早期 UI | =default-frame-alist= でツールバーオフ |

=straight-build-dir= は Emacs のバージョンを含みます（=build-32.0=）。異なる Emacs
バージョンのビルド生成物が混ざらないようにするためです。

=straight-built-in-pseudo-packages= に =org= を追加してあります。Org は Emacs 同梱
（9.8.7）ですが、=org-modern= 等が =Package-Requires= に =(org "9.x")= を宣言するため、
これを追加しないと straight が依存解決で org を clone してしまいます。

| ステップ | 内容 | 注意点 |
|----------+------+--------|

| §2 | straight ブートストラップ | 初回のみネットワークアクセス |
| §3 | =leaf= 初期化 | =leaf-keywords-init= 必須 |

| §8 | 組み込みポリシー | =use-short-answers=、=C-z= アンバインド |
| §9 | パーソナルオーバーレイ | =user.el= + =<login>.el= をロード。=my:modules-extra= を追記 |

*Org は組み込みを使う（straight では入れない）*

以前は §4 で =(straight-use-package 'org)= と =(require 'org)= を実行していましたが、
削除しました。Emacs 32.0.50 は Org 9.8.7 を同梱しており、本設定が使う Org 関連
（=ox-hugo=、=org-roam=、=org-modern=、=org-journal=、=org-appear=、=valign=、
=ob-mermaid=、=org-fold=）はいずれも組み込みで動きます。straight 版は "mixed Org version"
問題と、バージョンごとの再ビルドを持ち込むため採用しません。

なお Org はセクション §4 で =require= されなくなっただけでなく、orgx 層でも遅延ロード
されます（後述の orgx Layer を参照）。

* modules.el — Delete-Safety の実装

=modules.el= は設定システムの司令塔です。
一番大事なのは「1 本壊れても全体が止まらない」という delete-safety の保証で、
その核心が =condition-case= による各モジュールの隔離です。

[[file:svg/02_condition_case.svg]]

- =error= だけを捕まえます。=quit= (=C-g=) は意図的に伝播させます。
  ユーザーが中断したいときは中断できるようにしておく必要があるので。
- =my:modules-verbose= が非 nil なら各モジュールのロード時間をミリ秒単位で表示します。
- =my:modules-skip= にシンボルを入れておくと、そのモジュールを静かにスキップできます。
  たとえば =dev-docker= を使わないデバイスでは省略できます。
- =my:modules-extra= は =personal/user.el= が =setq= で追記します。正規リストの後ろに
  追加されます。=setq= が正しい理由はロード順にあり、personal オーバーレイが走る時点では
  =modules.el= の =defcustom= がまだ評価されていないためです（=defvar= の束縛だけが有効）。

#+begin_src emacs-lisp
;; modules.el の核心部分
(dolist (feature (append my:modules my:modules-extra))
  (unless (memq feature my:modules-skip)
    (let ((t0 (current-time)))
      (condition-case err
          (progn (require feature) t)
        (error
         (message "[modules] Failed to load %s: %s"
                  feature (error-message-string err))
         nil))
      (when my:modules-verbose
        (message "%-24s %s"
                 feature
                 (modules--format-seconds
                  (float-time (time-subtract (current-time) t0))))))))
#+end_src

** 2 チャンネルタイマー設計（現状は不活性）

スケジューラは 2 系統のタイマーを持つ設計になっています。

| タイマー | 実装 | 周期 | 用途 |
|----------+------+------+------|
| 定期タイマー | =(run-at-time 600 600 ...)= | 600 秒ごと | セッション定期メンテ |
| アイドルタイマー | =(run-with-idle-timer 120 t ...)= | 120 秒アイドル後 | バックグラウンド処理 |

ジョブは =make-hash-table :test 'eq= で管理され、=maphash= で走査するため中間リストの
アロケーションが不要です。

#+begin_quote
*現状:* このジョブスケジューラは機能的に不活性です。=core-session-register-job= の呼び出しが
設定全体に 1 つも無いため、ジョブテーブルは常に空で、2 本のタイマーは空のハッシュを走査し
続けます。スケジューラ自体を削除するか、将来のジョブ登録用の拡張ポイントとして温存するかは
未決です。将来ジョブを登録する場合の足場として、設計そのものは残してあります。
#+end_quote

* GC Strategy — 2 層の協調設計

GC 管理は 2 つのモジュールが役割を分担しています（起動ガードを含めれば 3 層）。

[[file:svg/04_gc_strategy.svg]]

| 層 | ファイル | タイミング | 内容 |
|----+----------+------------+------|
| 起動ガード | =early-init.el §8= | 起動時 | =file-name-handler-alist= を nil に退避。=after-init-hook= で復元 |
| 明示トリガー | =core-gc.el= | フォーカス喪失 / ミニバッファ終了 | =garbage-collect= を =condition-case= で安全に実行 |
| アイドル管理 | =gcmh= (=core-runtime.el=) | アイドル 5 秒後 | 16MiB 閾値。=leaf :custom= では式評価不可なので =:config= で =setopt= |

*フォーカス喪失時の GC*

=core-gc.el= はフォーカス喪失を契機に GC を走らせますが、=focus-out-hook= は Emacs 27.1 で
廃止されています。後継の =after-focus-change-function= を使い、=frame-focus-state= で
「フォーカスを失った」ことを判定してから GC します（この関数はフォーカス取得時にも呼ばれる
ため）。=after-focus-change-function= は「フック」ではなく単一の関数を保持する変数なので、
=add-hook= ではなく =add-function= で登録します。

#+begin_quote
*落とし穴:* =gcmh-high-cons-threshold= に =(* 16 1024 1024)= のような式を渡すとき、
=leaf :custom= に書くと式がリテラルとして格納されてしまいます。
=:config= 内で =(setopt gcmh-high-cons-threshold (* 16 1024 1024))= と書くのが正解です。
#+end_quote

* Auto-Save Subsystem — 保存サブシステム

=core-editing.el= の自動保存サブシステムは 2 つのトリガーと 4 段階の除外判定で構成されて
います。

[[file:svg/05_autosave_flow.svg]]

** 保存トリガー

| トリガー | 仕組み |
|----------+--------|
| アイドル ≥5 秒 | =auto-save-visited-mode= のタイマー。=core-autosave-interval= (defcustom, デフォルト 5) で調整 |
| フォーカス喪失 | =after-focus-change-function= → =core-editing--save-on-focus-out=（=frame-focus-state= で喪失を判定） |

*focus-out-hook を使わない理由*

以前は =focus-out-hook= と =after-focus-change-function= の両方に保存関数を登録していました。
しかし =focus-out-hook= は Emacs 27.1 で廃止されており、しかも両方に登録していたため、
フォーカスを失うたびに保存が二重に走っていました（=handle-focus-out= は =focus-out-hook= を
実行したうえで =after-focus-change-function= も呼ぶため）。加えて =after-focus-change-function=
はフォーカス取得時にも呼ばれるため、他アプリから Emacs へ戻るたびにも保存が走っていました。

現在は =after-focus-change-function= のみに登録し、=core-editing--save-on-focus-out= が
=(not (frame-focus-state))= で「喪失」を確認してから保存します。

** 除外 defcustom 変数

| 変数名 | デフォルト | 説明 |
|--------+------------+------|
| =core-autosave-exclude-extensions= | =gpg age enc pem= ... | 拡張子で除外 |
| =core-autosave-exclude-name-patterns= | =nil= | フルパスに対する正規表現リスト |
| =core-autosave-exclude-modes= | =nil= | 除外するメジャーモード一覧 |
| =core-autosave-exclude-predicates= | =nil= | =(buf) → bool= な関数リスト |

いずれも =personal/user.el= で =setopt= を使って上書きできます。

* auth Layer — 認証情報の解決フロー

認証まわりは 3 つのモジュール（=auth-core=、=auth-gpg=、=auth-keychain=）が連携します。
特に aidermacs の API キー取得は =after-init-hook= の発火順序に依存する繊細な設計です。

[[file:svg/06_auth_resolution.svg]]

** ~/.authinfo.gpg に必要なエントリ

#+begin_example
# Anthropic Claude API (utils-claude.el 用)
machine api.anthropic.com login apikey password sk-ant-...

# OpenRouter (aidermacs 優先バックエンド)
machine openrouter.ai login ac1965 password sk-or-...

# OpenAI (aidermacs フォールバック)
machine api.openai.com login org-ai password sk-proj-...
#+end_example

=login= フィールドは =:user= 引数に完全一致しないと =auth-source-search= が何も返しません。
不一致でもエラーにならず黙って nil を返すため、気づきにくいハマりどころです。
診断には =my/auth-gpg-verify= が使えます。

** macOS Keychain バックエンド

=auth-keychain= は =security(1)= に委譲する独自の =auth-source= バックエンドで、macOS 以外の
ホストでは不活性です。=auth-gpg= の後に =auth-sources= へ追加されるため、参照順序は
=~/.authinfo.gpg= → =~/.authinfo=（平文フォールバック）→ Keychain となり、GPG 暗号化ストアが
常に優先されます。有効・無効は =personal/device-darwin.el=（デバイス単位）と
=personal/<login>.el=（ログイン単位）で =auth-keychain-enable= を設定して制御します。

** after-init-hook の発火順序

=auth-gpg= が =dev-ai= よりも =modules.el= で先にロードされることを利用しています。

#+begin_example
auth-gpg ロード → after-init-hook に configure-sources を登録
dev-ai   ロード → after-init-hook に setup-api-key を登録
  ↓ after-init-hook 発火
  #1: auth-gpg--configure-sources → ~/.authinfo.gpg を auth-sources 先頭に追加
  #2: dev-ai--setup-api-key → auth-source-search が成功 ✓
#+end_example

=dev-ai= の =:init= で検索する旧実装ではこの順序が保証できず、常に "no key found" に
なっていました。API キーの参照は =after-init-hook= で行うのが正解です。

* dev Layer — LSP バックエンド非依存設計

[[file:svg/08_lsp_agnostic.svg]]

=dev-lsp-bridge.el= を =dev/= セクションの先頭に置くことで、
=ui-keymap.el= がバックエンドを知らなくても =dev-lsp-code-actions= などにキーをバインドできます。

これらは =my:modules= に含まれません。宣言元モジュールの =autoload= 経由で遅延ロードされます。

| モジュール | 宣言先 | トリガー |

| =dev-lsp-eglot= | =core-switches= | after-init 時に =(my:use-lsp 'eglot)= |

* orgx Layer — Org 遅延ロード / Auto-Tangle / Auto-Hugo

[[file:svg/10_orgtangle_hugo.svg]]

** Org の遅延ロード

orgx 層は、Org を起動時に読み込まないよう設計されています。以前は次の 5 経路が Org を
無条件にロードしていましたが、すべて遅延化しました。

| モジュール | 旧 | 現在 |
|------------+----+------|
| =orgx-core= | =(leaf org :require t)= | =:require t= を外す。=:custom= は Custom が先行登録、=:config= は =with-eval-after-load 'org= に包まれる |
| =orgx-export= | =(leaf ox :require t)= | 削除（ox は ox-hugo が require） |
| =orgx-export= | =(leaf ox-hugo :require t)= | =:require t= を外す |
| =orgx-export= | =(leaf ob-mermaid :require t)= | =with-eval-after-load 'org= で登録 |
| =orgx-auto-tangle= | トップレベル =(require 'org)= =(require 'ob-tangle)= | =declare-function= + 関数内 require |
| =orgx-auto-hugo= | トップレベル =(require 'ox-hugo)= | =declare-function= + 関数内 require |

さらに、既定構成で org-roam を起動時にロードしないため、=orgx-extensions= の org-roam
autosync タイマー（旧: 起動 5 秒後）を廃止し、対話コマンド =my/org-roam-autosync-enable= に
置き換えました。=org-roam-node-find= 等は autoload されるため、初回呼び出し時に一度だけ DB が
同期されます。=orgx-roam-ui= も既定の =my:modules-extra= から外してあります。

これにより、Org（および org-roam）は「=.org= を開く / Org のコマンドを起動する / org-roam の
コマンドを起動する / =my/org-roam-autosync-enable= を実行する」のいずれかまでロードされません。

** orgx-auto-tangle

=after-save-hook= をバッファローカルに登録しているので =README.org= 以外には影響しません。
再帰ループ防止のため =orgx-auto-tangle--in-progress= をバッファローカルフラグとして使い、
=ob-tangle= は tangle 直前に require します。

#+begin_src emacs-lisp
(declare-function org-babel-tangle "ob-tangle" (&optional arg target-file lang-re))

(defvar-local orgx-auto-tangle--in-progress nil)

(defun orgx-auto-tangle--maybe ()
  (when (and (orgx-auto-tangle--eligible-p)
             (not orgx-auto-tangle--in-progress))
    (require 'ob-tangle)
    (let ((orgx-auto-tangle--in-progress t)
          (org-confirm-babel-evaluate nil)
          (inhibit-modification-hooks t))
      (condition-case err
          (org-babel-tangle)
        (error (message "[auto-tangle] %s" err))))))
#+end_src

** orgx-auto-hugo

=after-save-hook= ではなく =org-capture-after-finalize-hook= を使っています。
キャプチャ編集中の自動保存で ox-hugo がトリガーされると未完成エントリがエクスポートされてしまうためです。
=org-note-abort= が非 nil なら (=C-c C-k= でキャンセルされた場合) エクスポートをスキップします。
=ox-hugo= はエクスポート直前に require し、起動時には読み込みません。

* ui-leader.el — M-SPC リーダーキー階層

[[file:svg/11_leader_keymap.svg]]

=M-SPC= を起点にサブマップが並んでいます。
元が =C-c SPC= (3 キー) だったのを =M-SPC= (2 キー) に変更して、

バインドには Emacs 29 以降の =keymap-set= API を用います。

* 命名規則と変数代入ポリシー

** 命名規則

| プレフィックス | スコープ | 例 |
|----------------+----------+----|
| =my/= | 公開インタラクティブコマンド | =my/eww-search=, =my/auth-gpg-verify= |
| =<module>-= | モジュール公開 API | =core-treesit-install-all=, =ui-font-apply= |
| =<module>--= | モジュール内部実装 | =core-native--collect=, =modules--require-safe= |
| =my:d:*= | パス / ディレクトリ変数 | =my:d=, =my:d:var=, =my:d:org= |
| =my:modules= | 正規リスト定数 | =defconst= で定義。変更禁止 |

禁止パターン: =my/<topic>--= の複合形。公開名前空間 (=my/=) とプライベートスコープ (=--=) を混ぜてはいけません。

** setopt vs setq

| 使うべき形 | 対象 | 補足 |
|------------+------+------|
| =setopt= | =defcustom= 変数 | セッターと型チェックを通す |
| =setq= | =defvar=, 内部状態変数 | セッターが不要な場合 |
| =setopt= (=:config= 内) | 実行時評価が必要な =defcustom= パス | =:custom= では式評価不可 |
| =setq-local= | バッファローカルな =defcustom= 上書き | Makefile の =indent-tabs-mode= など |
| =customize-save-variable= | セッションをまたいで永続化が必要な =defcustom= | =personal-messages-last-archived= など |

=setq= が正解な =defcustom= 例外:
=org-agenda-files=, =org-capture-templates=, =org-todo-keywords=, =org-refile-targets=, =org-roam-db-connector=。
バッククォートや実行時評価が =setopt= の型チェッカーと相性が悪いためです。

なお =personal/user.el= の =my:modules-extra= も =setq= で代入します。これは
=defcustom= 例外ではなくロード順の要請で、personal オーバーレイが走る時点では
=modules.el= の =defcustom= がまだ評価されておらず、=init.el= の =(defvar my:modules-extra nil)=
の束縛のみが有効なためです。

* Coding Rules R1–R8

** R1 — lexical-binding

全ファイルの 1 行目に =;; -*- lexical-binding: t; -*-= を必ず書きます。
ないとラムダが動的スコープで動いてクロージャのバグが出ます。
=Makefile= の =check-cookies= ターゲットが全生成物についてこれを検証します。

** R2 — =provide= シンボルとファイル名の一致

#+begin_src emacs-lisp
;;; core-treesit.el
(provide 'core-treesit)  ; ファイル名と一致させる
#+end_src

** R3 — 組み込みパッケージに =:straight nil=

#+begin_src emacs-lisp
(leaf saveplace :straight nil
  :pre-setq
  `((save-place-file . ,(concat no-littering-var-directory "saveplace"))))
#+end_src

Org も組み込みなので =orgx-core= は =(leaf org :straight nil)= と宣言します
（=:require t= は付けません。遅延ロードのため）。

** R4 — =leaf= キーワード順序

#+begin_example
:straight → :ensure → :after → :require →
:pre-setq → :custom → :bind → :hook → :init → :config
#+end_example

** R5 — 公開 =defun= のドキュメント文字列

公開 =defun= には必ずドキュメント文字列を書きます。インタラクティブコマンドは特に必須です。

** R6 — =setopt= / =setq=

上の「命名規則と変数代入ポリシー」セクション参照。

* 静的検査（Makefile）

品質チェックは =make lint= に集約されています。次の 3 つを順に実行します。

| ターゲット | 内容 |
|------------+------|
| =check-tangle= | =:tangle= を継承できない emacs-lisp ブロックを検出（見出しレベルの誤りを静的に捕捉） |
| =check-cookies= | 全 =.el= の 1 行目にある lexical-binding クッキーを検証 |
| =checkdoc= | 公開 =defun= のドキュメント文字列を検証 |

=check-tangle= は、=**** foo Design Notes= を =*** foo Design Notes= と 1 段間違えて
親の兄弟にしてしまい、配下の src ブロックが =:tangle= を失う事故（実際に複数回発生）を
静的に検出するためのものです。

| キーワード | 要点 |
|------------+------|
| Boot | 3 フェーズ決定論的。=my:modules= リストが唯一の起動順序定義 |
