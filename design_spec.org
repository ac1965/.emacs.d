* 起動シーケンス

** 3フェーズの全体像

Emacs の起動は内部的に 3 段階に分かれてる。

- *Phase 1 (early-init.el)* — フレームが描画される前の最初期処理。
  =package.el= を無効化して、ディレクトリを整えて、GC を一時的に緩める。
- *Phase 2 (init.el)* — =straight.el= と =leaf= をブートストラップして、
  モジュールシステムを起動する手前の下ごしらえをする。
- *Phase 3 (modules.el)* — 75 本のモジュールを決められた順番でロードする。
  1 本コケても残りの 74 本は続く（delete-safety、後述）。

[[file:svg/01_boot_flow.svg]]

*Org を組み込みに戻した理由（§4）*

以前は =init.el= で =(straight-use-package 'org)= していたが、これは撤去した。理由は次のとおり。

- 使っている Org 関連パッケージ（ox-hugo、org-roam、org-modern 等）はどれも組み込み Org で動く。bleeding-edge な Org を要求するものはない。
- straight 版だと「組み込み Org の autoload に一度でも触れたら壊れる」という "mixed Org version" 問題を抱え込む。決定論的起動という原則にとって負債でしかない。
- straight が依存関係解決の過程で Org を勝手に clone してしまう問題があったので、 =straight-built-in-pseudo-packages= に =org= を追加して対処済み（設定は =early-init.el= 側、straight のブートストラップより前である必要がある）。

* modules.el — Delete-Safety の実装

=modules.el= は設定システムの司令塔。
一番大事なのは「1 本壊れても全体が止まらない」という delete-safety の保証で、
その核心が =condition-case= による各モジュールの隔離。

[[file:svg/02_condition_case.svg]]

- =error= だけを捕まえる。 =quit= (=C-g=) は意図的に伝播させる。
  ユーザーが中断したいときは中断できるようにしておく必要があるので。
- =my:modules-verbose= が非 nil なら各モジュールのロード時間をミリ秒単位で表示する。
- =my:modules-skip= にシンボルを入れておくと、そのモジュールを静かにスキップできる。
  たとえば =dev-docker= を使わないデバイスでは省略できる。
- =my:modules-extra= は =personal/user.el= が追記する。正規リストの後ろに追加される。
  ちなみにこれは =setopt= ではなく =setq= で代入する。理由は、personal オーバーレイが走る時点では
  まだ =defcustom= が評価されていない（ =modules.el= のロードより前）ため、 =setopt= を使うと
  =void-variable= になってしまうから。

#+begin_src emacs-lisp
;; modules.el の核心部分
(dolist (feature (append my:modules my:modules-extra))
  (unless (memq feature my:modules-skip)
    (let ((t0 (current-time)))
      (condition-case err
          (progn (require feature) t)
        (error
         (message "[modules] Failed to load %s: %s"
                  feature (error-message-string err))
         nil))
      (when my:modules-verbose
        (message "%-24s %s"
                 feature
                 (modules--format-seconds
                  (float-time (time-subtract (current-time) t0))))))))
#+end_src

| タイマー | 実装 | 周期 | 用途 |

;; core-session-private.el より
(defun core-session--run-all ()
  "全ての有効ジョブを実行する。maphash で走査し中間リストを作らない。"
  (maphash (lambda (_name meta)
             (when (plist-get meta :enabled)
               (condition-case _err
                   (funcall (plist-get meta :fn))
                 (error nil))))
           core-session--jobs-table))
#+end_src

* GC Strategy — 3 層の協調設計

GC 管理は 3 つのモジュールが役割を分担している。

[[file:svg/04_gc_strategy.svg]]

| 層 | ファイル | タイミング | 内容 |
|----+----------+------------+------|
| 起動ガード | =early-init.el §8= | 起動時 | =file-name-handler-alist= を nil に退避。しきい値を一時的に 512MiB まで緩め、=after-init-hook= で 128MiB に戻す |
| 明示トリガー | =core-gc.el= | フォーカス変更 / ミニバッファ終了 | =garbage-collect= を =condition-case= で安全に実行 |
| アイドル管理 | =gcmh= (=core-runtime.el=) | アイドル 5 秒後 | 128MiB 閾値。=leaf :custom= では式評価不可なので =:config= で =setopt= |

#+begin_quote
*落とし穴:* =gcmh-high-cons-threshold= に =(* 128 1024 1024)= を渡すとき、
=leaf :custom= に書くと cdr 位置がそのままクォートされてリテラルのリストとして代入されてしまい、
=gcmh-mode= の内部で =Wrong type argument: integerp, nil= を引き起こす。
=:config= 内で =(setopt gcmh-high-cons-threshold (* 128 1024 1024))= と書くのが正解。
#+end_quote

* auth Layer — 認証情報の解決フロー

認証まわりは 3 つのモジュールが連携する。

[[file:svg/06_auth_resolution.svg]]

** ~/.authinfo.gpg に必要なエントリ

#+begin_example
# Anthropic Claude API (utils-claude.el 用)
machine api.anthropic.com login apikey password sk-ant-...

# OpenRouter (aidermacs 優先バックエンド)
machine openrouter.ai login ac1965 password sk-or-...

# OpenAI (aidermacs フォールバック)
machine api.openai.com login org-ai password sk-proj-...
#+end_example

=login= フィールドは =:user= 引数に完全一致しないと =auth-source-search= が何も返さない。
不一致でもエラーにはならず黙って空を返すだけなので、「キーを書いたのに認識されない」ときは
まずここを疑う。

** after-init-hook 頼みだけじゃない、二段構えの安全策

=auth-gpg= が =dev-ai= よりも先に =modules.el= でロードされ、両者とも =after-init-hook= に
処理を登録している——なので通常は「先に auth-gpg が =~/.authinfo.gpg= を =auth-sources= の
先頭に追加し、その後で dev-ai がキーを検索する」という順序で成功する。

#+begin_example
auth-gpg ロード → after-init-hook に configure-sources を登録
dev-ai   ロード → after-init-hook に setup-api-key を登録
  ↓ after-init-hook 発火
  #1: auth-gpg--configure-sources → ~/.authinfo.gpg を auth-sources 先頭に追加
  #2: dev-ai--setup-api-key → auth-source-search が成功 ✓
#+end_example

ただ、これだけだと =after-init-hook= 内でのフック登録順という、やや繊細な前提に依存してしまう。
なので =dev-ai--setup-api-key= 自身にも =dev-ai--ensure-authinfo-in-sources= を仕込んであり、
検索前に「 =auth-gpg-file= （未ロードなら =~/.authinfo.gpg= にフォールバック）が
=auth-sources= に入っているか」を自前でチェックして、無ければ先頭に追加してキャッシュを
クリアする。つまり *auth-gpg のロード順や実行順序がどうであれ* 、dev-ai 側だけで
検索が成功するように二重に保険を掛けてある。

* dev Layer — LSP バックエンド非依存設計

[[file:svg/08_lsp_agnostic.svg]]

=dev-lsp-bridge.el= を =dev/= セクションの先頭に置くことで、
=ui-keymap.el= がバックエンドを知らなくても =dev-lsp-code-actions= などにキーをバインドできる。

** autoload-only モジュール

これらは =my:modules= に含まれない。 =core-switches.el= 経由で、必要になったときだけ遅延ロードされる。

| モジュール | 宣言先 | トリガー |
|------------+--------+---------|
| =dev-lsp-eglot= | =core-switches= | after-init 時に =(my:use-lsp 'eglot)= |
| =dev-lsp-mode= | =core-switches= | after-init 時に =(my:use-lsp 'lsp)= |
| =ui-doom-modeline= | =core-switches= | after-init 時に =(my:use-ui 'doom)= |
| =ui-nano-modeline= | =core-switches= | after-init 時に =(my:use-ui 'nano)= |
| =ui-nano-palette= | =ui-theme= | =(require 'ui-nano-palette nil t)= |

=my:use-lsp= は =none= / =eglot= / =lsp= の 3 択、 =my:use-ui= は =none= / =doom= / =nano= の 3 択。
これらのモジュールは =leaf= 宣言で =:require t= を使ってはいけない。autoload 宣言が
load-path 上の可用性を保証してくれるので、そこで =:require t= を書くと遅延ロードの意味が無くなる。

* orgx Layer — Auto-Tangle & Auto-Hugo

[[file:svg/10_orgtangle_hugo.svg]]

orgx 層は全部で 11 モジュール。 =my:modules= に入ってる正規モジュールは 8 つで、残る 3 つ
（ =orgx-typography= / =orgx-brain= / =orgx-roam-ui= ）はオプション拡張として =my:modules-extra=
経由で足すかどうかを選ぶ形。 =orgx-roam-ui= は既定では外してある——理由は次の「Org の遅延ロード」参照。

** Org の遅延ロード

Org を組み込みに戻しただけでは、orgx 層に Org を無条件で読み込む経路が残っていた。
すべて遅延化してある。

- =orgx-auto-tangle= と =orgx-auto-hugo= は、トップレベルで =(require 'org)= / =(require 'ob-tangle)=
  / =(require 'ox-hugo)= を *しない* 。代わりに =declare-function= でバイトコンパイラ向けの宣言だけ
  済ませておき、実際の =require= は関数が実行されるタイミング（フック発火時）まで遅らせる。
- どちらのフックも、発火する時点で Org は必ずロード済みという前提が成り立つ
  （ =orgx-auto-tangle= は =org-mode-hook= 経由でしか登録されないし、 =orgx-auto-hugo= の
  =org-capture-after-finalize-hook= も同様）ので、この遅延化は安全。

ただし、既定構成では起動直後のアイドル時間に =orgx-extensions= が =org-roam-db-autosync-mode=
を有効化するため、結局 Org はそこそこ早めにロードされる。org-roam を使わない構成
（ =my:modules-skip= に =orgx-extensions= を追加）なら、Org を本当に使うまで完全に遅延できる。

** orgx-auto-tangle

=after-save-hook= をバッファローカルに登録しているので =README.org= 以外には影響しない。
再帰ループ防止のため =orgx-auto-tangle--in-progress= をバッファローカルフラグとして使っている。

#+begin_src emacs-lisp
(declare-function org-babel-tangle "ob-tangle" (&optional arg target-file lang-re))
(defvar-local orgx-auto-tangle--in-progress nil)

(defun orgx-auto-tangle--maybe ()
  (when (and (orgx-auto-tangle--eligible-p)
             (not orgx-auto-tangle--in-progress))
    (require 'ob-tangle)
    (let ((orgx-auto-tangle--in-progress t)
          (org-confirm-babel-evaluate nil)
          (inhibit-modification-hooks t))
      (condition-case err
          (org-babel-tangle)
        (error (message "[orgx-auto-tangle] error: %s" (error-message-string err)))))))
#+end_src

#+begin_quote
*Emacs 31 での追加対策:* =README.org= は約 1.3 万行、tangle 先はおよそ 90 ファイル。保存が
連続すると（ =auto-save-visited-mode= に加えて明示の =C-x C-s= が重なるなど）全 tangle が
2 回走ってしまうことがあった。 =timeout.el= の =timeout-throttle= で一定間隔内の連続呼び出しを
1 回にまとめている。実行中フラグ自体は非再入性の担保として引き続き必要（スロットリングは
頻度制限であって非再入性の保証ではないため）。
#+end_quote

** orgx-auto-hugo

=after-save-hook= ではなく =org-capture-after-finalize-hook= を使う。
キャプチャ編集中の自動保存で ox-hugo がトリガーされると未完成エントリがエクスポートされて
しまうため。 =org-note-abort= が非 nil なら（ =C-c C-k= でキャンセルされた場合）エクスポートを
スキップする。 =ox-hugo= のロードも同様に、トップレベルの =require= ではなく関数内部まで遅延している。

* ui-leader.el — M-SPC リーダーキー階層

[[file:svg/11_leader_keymap.svg]]

=M-SPC= を起点に 13 個のサブマップ（ =a b c e g h m o p q s t w= ）が並んでる。

* 命名規則と変数代入ポリシー

** 命名規則

| プレフィックス | スコープ | 例 |
|----------------+----------+----|
| =my/= | 公開インタラクティブコマンド | =my/eww-search=, =my/auth-gpg-verify= |
| =<module>-= | モジュール公開 API | =core-treesit-install-all=, =ui-font-apply= |
| =<module>--= | モジュール内部実装 | =core-native--collect=, =modules--require-safe= |
| =my:d:*= | パス / ディレクトリ変数 | =my:d=, =my:d:var=, =my:d:org= |
| =my:modules= | 正規リスト定数 | =defconst= で定義。変更禁止 |

禁止パターン: =my/<topic>--= の複合形。公開名前空間 (=my/=) とプライベートスコープ (=--=) を混ぜてはいけない。

** setopt vs setq

| 使うべき形 | 対象 | 補足 |
|------------+------+------|
| =setopt= | =defcustom= 変数 | セッターと型チェックを通す |
| =setq= | =defvar=, 内部状態変数 | セッターが不要な場合 |
| =setopt= (=:config= 内) | 実行時評価が必要な =defcustom= パス | =:custom= では式評価不可 |
| =setq-local= | バッファローカルな =defcustom= 上書き | Makefile の =indent-tabs-mode= など |
| =customize-save-variable= | セッションをまたいで永続化が必要な =defcustom= | =personal-messages-last-archived= など |

=setq= が正解な =defcustom= 例外は、次の 3 つに限定される: =org-todo-keywords= 、
=org-refile-targets= 、 =org-capture-templates= 。いずれもバッククォートや =(function ...)=
による実行時評価を含む値を持つため、 =setopt= の型チェッカが値の完全な評価前に誤検知してしまう。
=:custom= や =:pre-setq= に移すこともできない。

#+begin_quote
*注意（以前との違い）:* =org-roam-db-connector= は setq 例外ではなく、現在は普通に =setopt=
で設定している（Emacs 31.0.50 で =sqlitep= の挙動が変わったことへの対応として、 =:config= 内で
=setopt org-roam-db-connector ...= と明示的に固定している）。また =org-agenda-files= は setq で
代入するが、これは型チェッカ回避のためではなく、「モジュールロード時は inbox.org だけの
フォールバック値、 =after-init-hook= で完全な一覧に差し替える」という 2 フェーズ構成のための
別理由による setq なので、上の 3 つの例外とは性質が違う。
#+end_quote

* Changelog について — Fix ID 方式は廃止済み

以前はここに =Fix A→CJ= のようなコード付きの修正一覧を置いていたが、README.org 側で
Changelog を一度リセットし、コード中のコメントや散文からも Fix ID 参照（ =[Fix BZ-1]= や
=[R6 FIX]= など計 74 箇所）をすべて削除している。以後、コードから Changelog のエントリを
参照する運用はやめて、変更理由は該当箇所の Commentary に直接書き、変更履歴そのものは
Git のコミットログで追う方針になった。このドキュメントも同じ方針に合わせて、以下は
Fix ID を振らない「最近やったこと」のダイジェストにとどめる。

| カテゴリ | 内容 |
|----------+------|
| 未参照コードの削除 | 到達可能性解析で 426 定義中 30 個の到達不能コードを検出・削除（-313 行）。4 モジュールを =my:modules= から除去 |
| Org を組み込みに戻す | straight 版 Org の導入をやめ、"mixed Org version" 問題と再ビルドコストを解消 |
| Org の遅延ロード | =orgx= 層の 5 箇所の即時ロード経路をすべて遅延化 |
| org-roam-ui の local-repo 衝突解消 | =simple-httpd= と =web-server= の straight local-repo 名衝突を分離して修正 |
| straight が組み込み Org を clone する問題 | =straight-built-in-pseudo-packages= に =org= を追加して解決 |
| *scratch* の flymake 警告 | =trusted-content= 対応。=*scratch*= では flymake を無効化する方向で対処 |
| org-roam を起動時にロードしない構成へ | autosync の起動時アイドルタイマーを撤去し、コマンド起動時のみに変更 |
| core-editing のクォート漏れ修正 | =show-paren-not-in-comments-or-strings= に渡す値のクォート漏れによる起動時エラーを修正 |
| 効かない Org 強調記法の検出 | =make check-emphasis=（Python 実装）を追加。全角文字に接した強調記法の見落としを検出 |
| Emacs 30/31 の廃止シンボル追随 | 1,717 件の廃止シンボルと突き合わせ、実際に使用していた 5 件を修正（=focus-out-hook= 等） |
| tangle 先を継承できない src ブロックの検出 | =make check-tangle= を追加。見出しレベルのズレによる静かな tangle 漏れを検出 |
| ドキュメントとコードの整合 | Layer Definitions・Directory Layout 等、実コードと矛盾していた記述を洗い出して修正 |
| Changelog のリセット | Fix ID 方式を廃止。以後は Git のコミットログで履歴を追う |
| 日本語化 | README.org の散文とコード内コメントを日本語化 |

* リファクタリングしたい場合のプロンプト

このドキュメント（ =design_spec.org= ）を =README.org= に合わせて更新したいときは、
だいたい次のようなプロンプトを投げればいい。

#+begin_quote
README.org に整合するように design_spec.org をリファクタリングして。
フランクな文体で。README.org 側で仕様や実装が変わっている箇所（モジュール構成、defcustom の
デフォルト値、トリガーの仕組み、Changelog の運用方針など）があれば design_spec.org 側を
正として README.org に合わせて直して。差分がある箇所は「以前との違い」がわかるように
補足してほしい。最後に scripts/check_emphasis.py のロジックで design_spec.org 自体も検査して、
全角文字に接して効かなくなっている =verbatim= / ~code~ / *bold* があれば直して。
#+end_quote

やってることとしては、README.org を =grep= / =sed= で該当セクション（モジュール一覧、
early-init.el / init.el のセクション番号、defcustom のデフォルト値、Changelog の運用方針など）
を実際に確認し、design_spec.org の記述とズレていないか 1 項目ずつ突き合わせる、という地道な作業。
README.org は 1 万行を超えるので、全文を読み込ませるより「この節とこの節を確認して」と
的を絞ったほうが速い。

最後の =check_emphasis.py= のくだりは、フランクな文体に書き換える過程で日本語の地の文が
増えたぶん、和文と =/~/*= が直接隣接する箇所が新たに生まれやすいので、忘れず流すためのステップ。
=check_emphasis.py= は元々 =README.org design_spec.org= のように複数ファイルを渡せる作りに
なっていて、design_spec.org も検査対象として想定されている。ただしこのスクリプトは *検出だけ*
で自動修正はしてくれない。=scripts/reload.py= も同様で、こちらは =clean → tangle →
check-cookies= の 3 ステップしかやらず、強調記法のチェックは含まれていない。なので今回は
Python で検出ロジックをそのまま動かし、違反箇所に半角スペースを 1 個差し込む形で機械的に
直した（本文の意味は変えず、全角の括弧・読点と記法の間に半角スペースを入れただけ）。
